CPU Scheduling:
The Art of Fairness & Speed

Every time you open an app, your operating system quietly makes thousands of scheduling decisions per second. NexOS lets you see exactly how that works — and why the choice of algorithm matters enormously.

Real-World Analogy Think of CPU scheduling like triage at a hospital emergency room. The scheduler is the triage nurse — deciding who gets treated first based on urgency (priority), how long they've waited (fairness), or how quickly they can be helped (burst time). A poor policy means critical patients wait too long. A smart one saves lives — and in computing, saves milliseconds that add up to billions in cost.

Why Does This Matter?

Modern computers run hundreds of processes simultaneously, but most CPUs have only a handful of cores. The scheduler decides which process gets CPU time at each moment. A bad scheduler wastes cycles, makes apps feel sluggish, and in real-time systems like aircraft avionics or medical devices, can cause catastrophic failures.

Efficient scheduling means fewer servers are needed in data centres, which directly translates to lower energy costs and carbon emissions. A 10% improvement in CPU utilisation at Google's scale saves hundreds of millions of dollars annually.

Where It's Used CPU scheduling runs inside Linux, Windows, and macOS kernels today. Cloud providers like AWS and Google Cloud use scheduling principles at massive scale. Real-time schedulers run inside aircraft flight computers, pacemakers, surgical robots, and automotive ECUs — systems where a missed deadline isn't a slowdown, it's a failure.

Key Concepts

Before diving into algorithms, understand the vocabulary. These terms appear in every scheduling textbook and real OS implementation.

Burst Time
How long a process needs the CPU. Short = I/O-bound. Long = CPU-bound.
Arrival Time
When a process enters the ready queue. Processes don't all arrive simultaneously in real systems.
Turnaround Time
Finish time − arrival time. Total time a process spends in the system from start to finish.
Waiting Time
Time spent in the ready queue not executing. The cost of being preempted or queued behind others.
Response Time
Time from arrival to first CPU execution. Critical for interactive applications where users notice lag.
Starvation
When a process waits indefinitely because higher-priority processes keep arriving and cutting in.
Preemption
The OS forcibly stopping a running process to give the CPU to another. Not all algorithms allow this.
Context Switch
Saving and restoring process state when the CPU switches between processes. Has overhead cost.
Throughput
Number of processes completed per unit time. Higher = more efficient use of the CPU.
Fairness Index
Measure of how evenly waiting time is distributed. High fairness = no process is heavily disadvantaged.
Time Quantum
In Round Robin, the maximum time a process can run before being preempted. Tuning this is critical.
Deadline
A hard or soft time constraint on when a process must finish. Missing it may be unacceptable in RT systems.

Scheduling Algorithms

Six fundamental approaches — each with different tradeoffs between fairness, throughput, response time, and implementation complexity.

FCFS
First Come, First Served
The simplest approach. Processes execute in arrival order, no preemption. Easy to implement but vulnerable to the Convoy Effect.
PreemptiveNo
Starvation riskNone
Best forBatch, uniform workloads
SJF
Shortest Job First
Picks the process with the shortest burst. Optimal for minimising average waiting time, but requires knowing burst times ahead of time.
PreemptiveNo
Avg WaitOptimal (non-preemptive)
Best forKnown, varied burst times
SRTF
Shortest Remaining Time First
Preemptive SJF. If a new process arrives with a shorter remaining time than the current one, it preempts immediately. Provides theoretical minimum average wait.
PreemptiveYes
Avg WaitTheoretical minimum
Best forMixed burst sizes
Round Robin
Time-Slice Scheduling
Each process gets a fixed time slice (quantum). After expiry, preempted and the next process runs. Fairest algorithm — no starvation. Response time depends heavily on quantum size.
PreemptiveYes
Starvation riskNone
Best forInteractive, time-sharing systems
Priority
Priority-Based Scheduling
Processes run in order of assigned priority. Higher-priority processes preempt lower ones. Can lead to starvation without aging mechanisms.
PreemptiveOptional
Starvation riskHigh
Best forReal-time, deadline-driven systems
MLFQ
Multi-Level Feedback Queue
Multiple queues with decreasing priority and increasing quantum. New processes start at the top; CPU hogs get demoted. Used in modern general-purpose OSes. Adapts to process behaviour without prior knowledge.
PreemptiveYes
Starvation riskLow
Best forGeneral-purpose OS, unknown workloads

Edge Cases & Classic Problems

Load these guided scenarios to observe notorious scheduling pathologies in action. Each reveals a fundamental weakness of certain algorithms.

⚡ The Convoy Effect

One long process holds the CPU while all short processes wait behind it — like a slow truck blocking a highway. Classic FCFS failure. Try it with FCFS, then switch to SJF.

💀 Process Starvation

A low-priority process never gets CPU time because high-priority processes keep arriving. Switch to Priority scheduling and watch LoPri wait indefinitely.

⏱ Quantum Sensitivity

Too-small quantum = excessive context switches. Too-large = degrades to FCFS. Load this workload then use the What-If Simulator on the Tradeoffs tab.

📊 Mixed Workload

A realistic mix of I/O-bound and CPU-bound processes. Compare how MLFQ vs Round Robin handles the heterogeneity differently in the Compare tab.

⚠ NexOS Starvation Detection NexOS automatically flags starvation when a process waits more than 3× its burst time. A red banner appears during simulation. Real OSes use "aging" — gradually boosting the priority of waiting processes — to prevent this. NexOS simulates this detection but not the fix, so you can see the problem clearly.

Real World Applications

CPU scheduling isn't academic — it's running right now inside every device you own, every server in every cloud, and every vehicle on the road.

🐧
Linux CFS
The Completely Fair Scheduler uses a red-black tree sorted by "virtual runtime" to ensure every process gets a mathematically equal CPU share. It's MLFQ taken to its logical extreme.
🪟
Windows NTFS
32 priority levels, multilevel queues. The foreground window gets a priority boost so your active app always feels responsive — a practical Priority scheduling tweak.
☁️
Cloud Computing
AWS and Google Cloud schedule millions of virtual machines using variants of these algorithms. Poor scheduling wastes millions in hardware and energy costs.
✈️
Avionics
Flight computers use Rate-Monotonic Scheduling — a variant of Priority where higher-frequency tasks get higher priority. Missing a deadline isn't a slowdown; it's a safety incident.
🏥
Medical Devices
Pacemakers and infusion pumps run RTOS schedulers guaranteed to execute within microseconds. Priority and deadline scheduling is what keeps them safe.
📱
Mobile OSes
iOS and Android extend MLFQ with power-awareness. Background tasks are deprioritised to save battery, while the UI render thread always runs first for a smooth 60 fps.
The Bottom Line The algorithms you're simulating in NexOS are the direct ancestors of the schedulers running inside Linux, Android, iOS, Windows, and every cloud server on Earth. Understanding them isn't just academic — it's foundational computer science that every engineer who works with performance-sensitive systems needs to know.
